# Git Learning Project

## Purpose
This is an educational project where we progressively build a web application from a single HTML file to a full-stack, multi-service cloud deployment. The primary goal is learning modern CI/CD practices, git workflows, and web architecture through hands-on experience.

## Current Phase
**Phase 1: Single HTML file + Git fundamentals**

## Key Principles
- **Learning first**: Every change is an opportunity to explain the "why". Don't rush.
- **Incremental complexity**: One concept at a time. Never skip ahead.
- **Real-world workflow**: Follow the branch-commit-PR-merge cycle even for small changes.
- **Explain every command**: The user is learning git and CI/CD from scratch through practice.

## Workflow for Every Change 
1. Plan — explain what we're adding and why
2. Branch — create a feature branch (once git is initialized)
3. Build — write the code, explaining each step
4. Test — verify it works locally
5. Review — review changes with `git diff`, write meaningful commits
6. Merge — merge to main (later: via PR)
7. Learn - at every step, be as education as possible to explain the task at hand, where it fits in the process, and explain best practices.

## Conventions
- Commit messages: conventional commits format (feat:, fix:, docs:, chore:)
- Branch names: `phase-N/short-description` (e.g., `phase-1/add-index-html`)
- Keep explanations concise but thorough on the "why"
- Always check ROADMAP.md for current phase and progress
